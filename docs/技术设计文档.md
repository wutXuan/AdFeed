# AI 广告流项目技术文档

## 1. 项目概述

该项目是一个 Android 原生广告信息流示例项目。用户可以在不同频道之间切换，通过标签筛选内容，下拉刷新、上拉加载更多，进入详情页，点赞、收藏、分享，同时系统会统计曝光和点击，并用大语言模型补全广告摘要、推荐理由、标签和自然语言搜索结果。

从工程结构上看，项目采用比较朴素但边界清楚的端侧架构：

- UI 层负责展示和交互，例如 `FeedFragment`、`AdFeedAdapter`、详情页和搜索弹窗。
- `FeedViewModel` 保存频道、标签、滚动位置等页面状态，把用户动作转发给数据层。
- `AdRepository` 是核心数据入口，统一协调 SQLite、本地种子广告、分页、埋点、AI 元数据补全和搜索。
- `AiRepository` 封装模型请求、输出解析、fallback和搜索预筛选。
- `data/local` 中的 SQLite 表负责持久化广告内容、AI 补全结果和 telemetry 明细。

我在设计时真正纠结的点有五个：广告卡片样式如何动态化、列表怎样在刷新和滚动时保持稳定、AI 输出不可控时如何不影响主流程、如何计算曝光合理、埋点事件是否需要引入真正的消息队列。下面按这几个问题展开。

## 2. 技术难点与整体思考

广告信息流看起来像一个列表，但实际难点比普通列表多。第一，广告卡片不是同一种样式：有大图、小图、视频，不同卡片的布局、媒体控件和交互状态都不同。第二，信息流会频繁刷新、筛选、加载更多，如果每次都整页重绘，用户会明显感到跳动。第三，AI 输出有价值，但模型返回慢、格式不稳定、甚至没有 API Key 时完全不可用，所以它不能成为列表可展示的前置条件。第四，曝光统计如果太宽松会虚高，如果太严格又会漏算，需要一个清楚、可解释的统计口径。第五，点赞、收藏、曝光、点击这些行为本质上都是事件，如果未来要上传服务端，最好不要和 UI 操作强耦合，但当前又只是 demo，没有必要引入完整服务端消息系统。

因此，项目最终采用的原则是：主链路优先稳定可用，AI 和统计能力以渐进增强的方式叠加。广告基础数据先从本地 SQLite 和种子数据加载，列表先展示出来；缺少 AI 信息的广告再异步补全；用户滚动过程中尽量减少重复计算和重复写入；曝光必须满足停留时间和可见比例两个条件。

## 3. 卡片动态化方案

### 难点

广告卡片的动态化不是简单地把字段填进 TextView。大图广告强调视觉面积，小图广告强调标题和摘要的横向扫描，视频广告还要处理播放、暂停、静音、播放器释放。如果把所有逻辑塞进一个布局，短期能跑，但后面每加一种样式都会让布局和绑定逻辑变得越来越难维护。

### 方案对比

第一种方案是“一个大布局硬塞所有样式”。也就是所有 item 共用一个 XML，通过控制 View 的显示隐藏来切换大图、小图、视频。这个方案实现最快，但问题很明显：布局层级会变深，很多 View 在某些类型下永远不可见；绑定时要处理大量 if/else；视频卡片的生命周期也容易和普通图片卡片混在一起。

第二种方案是“完整 DSL 动态渲染”。例如由后端下发卡片结构、组件类型、样式参数，端上解析后动态搭建 View。这是生产广告系统常见方向，扩展性强，但它需要稳定的协议、组件白名单、样式约束、容错机制和灰度能力。对于这个项目来说，它会把主要复杂度转移到协议设计和动态渲染框架上，超过了示例项目的规模。

第三种方案是“类型字段 + RecyclerView viewType + 多布局”。广告数据中保留 `type`，例如 `large_image`、`small_image`、`video`；`AdFeedAdapter.getItemViewType()` 根据 type 返回不同 viewType；`onCreateViewHolder()` 再加载对应的 item layout。这个方案没有 DSL 那么灵活，但它足够清晰，能覆盖当前业务需要，也方便以后新增一种卡片类型。

### 最终方案与原因

项目选择第三种方案。`AdFeedAdapter` 中通过 `AdItem.TYPE_VIDEO`、`AdItem.TYPE_SMALL_IMAGE` 区分卡片类型，并分别加载 `item_ad_video.xml`、`item_ad_small_image.xml`、`item_ad_large_image.xml`。普通图片卡片复用 `BaseHolder` 的绑定逻辑，视频卡片继承 `BaseHolder`，只额外处理 `PlayerView`、播放按钮、静音按钮和回收逻辑。

这样设计的原因是边界比较自然：标题、品牌、摘要、标签、点赞、收藏、分享这些能力属于所有广告卡片，可以放在基础 Holder；视频播放是特殊能力，只放在 `VideoHolder`。以后如果要加横滑商品卡、优惠券卡或直播卡，也可以继续增加 viewType，而不需要重写整个列表。

## 4. 列表性能策略

### 难点

信息流的性能问题通常不是某一个点造成的，而是多个小问题叠加：刷新时列表跳回顶部、点赞后整页闪烁、加载更多重复触发、视频播放器过多占资源、AI 补全后又导致列表刷新。项目虽然数据量不大，但仍然按真实信息流的思路处理这些问题。

### 方案对比

一种做法是使用普通 `RecyclerView.Adapter`，每次数据变化后调用 `notifyDataSetChanged()`。这很直接，但代价是所有可见 item 都会重新绑定，局部状态容易闪烁，用户刚点完赞或者刚滚动回来时体验不稳定。

另一种做法是引入分页库，例如 Paging 3。它适合真实后端分页和大规模数据加载，但当前项目的数据源主要是本地 SQLite 和 mock 数据，分页逻辑相对简单。如果为了少量本地数据引入完整 Paging 链路，文档和代码都会变重。

最终项目采用轻量分页配合 `ListAdapter + DiffUtil`。`AdRepository` 中 `PAGE_SIZE` 为 6，每次按频道和标签查询一页；`FeedFragment.maybeLoadMore()` 在最后可见项接近列表底部，也就是距离末尾 2 个 item 以内时触发加载更多。`ListAdapter` 负责提交新列表，`DiffUtil` 通过广告 id 判断是否同一项，再通过 `AdItem.equals()` 判断内容是否变化。

### 最终策略

列表性能主要由以下几层共同保证：

- 分页加载：首屏只取 6 条，滚动接近底部再加载下一页，减少一次性绑定和图片加载压力。
- 局部刷新：`ListAdapter + DiffUtil` 避免每次点赞、收藏、曝光计数或 AI 元数据更新后整页刷新。
- 滚动状态保存：`FeedViewModel` 用“频道 + 标签”作为 key 保存 `RecyclerView` 滚动状态，切换频道、筛选标签、进入详情返回时尽量恢复原位置。
- 加载更多去重：`AdRepository` 使用 `AtomicBoolean loadingMore` 避免快速滚动时重复发起加载更多。
- 视图复用：RecyclerView 使用独立 `RecycledViewPool`，滑出屏幕的卡片可以复用，降低 View 创建频率。
- 视频资源控制：`PlayerPool` 最多保留 2 个 `ExoPlayer`，视频卡片回收时释放 active player 到空闲池，避免每个视频 item 都创建播放器。

这个方案的好处是实现成本和项目规模匹配。它没有把示例项目做成大型分页框架，但已经解决了用户能直接感受到的卡顿、跳动和资源浪费问题。

## 5. AI 输出约束与缓存策略

### 难点

AI 能力在这个项目里有两个用途：一是给广告生成摘要、标签、推荐理由和轻松文案，二是根据用户自然语言搜索广告。难点在于模型输出不是稳定接口：它可能返回 markdown 代码块，可能字段缺失，可能返回不合法 JSON，也可能因为没有 API Key 或网络异常完全不可用。

如果 UI 等待 AI 成功后才展示广告，用户会看到空白或加载卡住；如果完全相信 AI 输出，又容易把异常文本展示给用户。因此设计目标是：AI 可以提升体验，但不能拖垮主流程。

### 方案对比

第一种方案是端上直接请求模型并展示原文。这最省事，但风险最高。模型如果返回解释性文字、markdown 或多余内容，UI 解析不了；模型如果生成过长文案，也会破坏卡片布局。

第二种方案是把 AI 全部放到服务端，由服务端做提示词、解析、审核和缓存。生产环境更推荐这种方式，因为 API Key 不会暴露在客户端，也更容易统一治理。但当前项目是端侧示例，如果引入服务端，会让运行门槛明显提高。

第三种方案是端上请求兼容 Chat Completions 风格的接口，但通过提示词、解析、fallback 和本地缓存降低不确定性。项目最终采用这个方案。

### 输出约束

`AiRepository.generateAdMeta()` 的 system prompt 明确要求“只返回 JSON，不要 markdown”，字段限定为 `summary`、`tags`、`reason`、`playfulCopy`，其中 `tags` 要求 3 到 5 个中文短标签。搜索场景也要求只返回 JSON，字段为 `results`，每项包含 `id`、`reason`、`matchedTags`。

即便如此，代码仍然没有假设模型一定听话。解析前会调用 `cleanJson()` 去掉可能出现的 markdown 代码块；解析时用 Gson/JsonParser 读取字段；字段为空时使用默认值；解析失败、网络失败、响应体为空或状态码异常时都会走本地 fallback。

### 缓存与降级

AI 缓存分两层：

- 请求去重缓存：`AdRepository` 用 `aiInFlight` 记录正在生成 AI 元数据的广告 id。如果同一批列表重复触发 `ensureAiForAds()`，已经在请求中的广告不会再次发起模型调用。
- 持久化缓存：AI 生成成功后，通过 `AdDao.updateAiMeta()` 把 `summary`、`tags`、`aiReason`、`playfulCopy` 写入 SQLite 的 `ads` 表。下次列表查询时，`AdItem.needsAiMeta()` 会判断这些字段是否已存在；如果已经存在，就不会再次请求。

无 API Key 时，项目不会请求远程模型，而是直接使用 `fallbackMeta()` 和 `fallbackSearch()`。这样做的好处是项目在没有任何外部服务的情况下仍然可以完整运行，AI 区域也不会空着。搜索场景还会先做本地预筛选：`prefilter()` 根据标题、品牌、描述、摘要和标签给候选广告打分，只把较相关的 16 条交给模型，减少 token 成本和模型误选范围。

这个策略的核心取舍是：端上 AI 不是生产最优解，但通过强约束、解析保护、fallback 和 SQLite 缓存，可以在示例项目里实现比较稳的用户体验。

## 6. 曝光统计口径

### 难点

曝光统计最容易出现“看起来有数据，实际不可信”的问题。用户快速滑过一张卡片，是否算曝光？卡片只露出一点点，是否算曝光？同一张广告在当前页面来回滑动，是否重复计数？这些口径如果不提前定义，后续点击率、转化率都会被污染。

### 方案对比

最宽松的方案是 item 一绑定就算曝光，也就是 `onBindViewHolder()` 中直接打点。它实现简单，但严重虚高：很多绑定出来的 item 用户可能根本没有看见，尤其 RecyclerView 会预取和复用。

更严格的方案是用可见面积、停留时长、页面生命周期和服务端去重综合判断。这更接近生产系统，但需要更复杂的计时器、可见面积计算、用户会话标识和服务端聚合。

项目采用一个中间口径：滚动后延迟检测，满足一定可见比例才算曝光，并在当前 Fragment 生命周期内对同一广告去重。

### 最终口径

当前曝光统计口径是：

- 每次滚动或滚动停止后，调用 `scheduleExposureCheck()`。
- 如果已有等待中的曝光检测任务，会先取消旧任务。
- 新任务延迟约 1 秒执行，表示广告需要在屏幕上停留一段时间。
- `trackVisibleExposure()` 遍历当前可见范围内的 item。
- 当卡片可见高度达到自身高度的 50% 及以上时，认为曝光成立。
- `exposedIds` 记录当前 Fragment View 生命周期内已经曝光过的广告 id，同一生命周期内同一广告只计一次。

曝光成立后，`FeedViewModel.trackExposure()` 转发到 `AdRepository.trackExposure()`。数据层会做两件事：一是调用 `AdDao.incrementExposure()` 增加 `ads.exposureCount`，用于卡片上的展示计数；二是插入一条 `telemetry` 明细，事件类型为 `exposure`，包含广告 id、频道、位置、时间戳等信息。

这个口径不等同于生产广告计费口径，但它清楚、可解释，并且比“绑定即曝光”可靠很多。它避免了快速滑过导致的明显虚高，也避免了同一次进入页面来回滚动造成重复计数。

## 7. 模拟消息队列设计

### 难点

项目里有很多用户行为事件，例如曝光、点击、点赞、收藏和分享。它们和普通 UI 状态不太一样：用户点了赞，界面要立刻反馈；但埋点事件更像一条“消息”，只要可靠记录下来，后续可以异步分析、上传或消费。如果把埋点上报直接写在 UI 点击逻辑里，页面代码会变重；如果一开始就引入 Kafka、Redis Stream 这类真正的消息队列，又会把一个端侧 demo 做成服务端工程。

所以这里的核心取舍是：需要保留“事件进入队列、异步处理、后续可扩展”的思想，但不引入真正的服务端消息中间件。

### 方案对比

第一种方案是 UI 事件发生后直接同步处理。例如点赞时直接更新数据库、直接插入埋点，甚至未来直接请求服务端接口。这个方案最简单，但 UI 和数据处理链路耦合太紧。一旦埋点上传失败、网络慢或后续要做批量上报，页面层就会被迫感知这些复杂性。

第二种方案是接入 Redis Stream。Redis Stream 支持追加消息、消费组和消息确认，适合做轻量事件流。但它需要一个 Redis 服务端、连接管理、鉴权、消费端和消息清理策略。更关键的是，Android 客户端不应该直接连 Redis：一方面会暴露内网地址和凭据，另一方面移动网络不稳定，客户端直连消息中间件的可靠性和安全性都不好。

第三种方案是接入 Kafka。Kafka 适合高吞吐、可持久化、多消费者的日志型消息系统，生产广告平台常用它承接曝光、点击、转化等海量事件。但 Kafka 的代价也更重：需要 broker 集群、topic 规划、分区、副本、consumer group、监控和运维。这个项目没有服务端，也没有海量事件吞吐需求，引入 Kafka 会让技术重点从 Android 信息流变成中间件搭建，明显不符合 demo 的边界。

第四种方案是本地模拟消息队列。也就是在端侧把事件封装成统一的 `TelemetryEvent`，由 `AdRepository` 作为事件生产入口，把事件异步写入 SQLite 的 `telemetry` 表。这里的 `telemetry` 表可以理解成一个本地消息缓冲区：UI 不直接关心后续怎么消费，只负责触发行为；Repository 把行为转成事件；DAO 负责持久化事件。

### 最终方案与原因

项目最终采用第四种方案：用本地 telemetry 明细表模拟消息队列，而不是接入 Redis Stream 或 Kafka。

具体流程是：

- UI 层触发行为，例如曝光、点击、点赞、收藏或分享。
- `FeedViewModel` 把动作转发给 `AdRepository`。
- `AdRepository` 创建 `TelemetryEvent`，统一记录 `adId`、`eventType`、`channel`、`position`、`extra` 和 `timestamp`。
- `AdDao.insertTelemetry()` 把事件写入 SQLite 的 `telemetry` 表。
- 数据写入运行在 Repository 的后台线程中，不阻塞用户点击和列表滚动。

这样做有三个原因。

第一，它保留了消息队列的核心思路。事件不是散落在 UI 代码里随手处理，而是统一进入 telemetry 明细表。后续如果要做批量上传、失败重试或本地分析，可以从这张表继续扩展。

第二，它符合 demo 的运行条件。项目不需要额外启动 Redis、Kafka 或后端服务，下载后只要 Android 工程能运行，就能看到完整的曝光、点击和互动事件记录。

第三，它避免了客户端直连中间件的风险。真实生产环境中，移动端通常应该把埋点发给服务端网关或日志接口，再由服务端写入 Kafka、Redis Stream 或其他消息系统，而不是让 App 直接连消息队列。

### 后续演进

如果项目从 demo 演进到真实投放系统，可以把当前模拟队列平滑升级：

- 端侧继续保留 `TelemetryEvent` 作为统一事件模型。
- SQLite `telemetry` 表可以升级为本地待上传队列，增加上传状态、重试次数和批次号。
- 服务端提供埋点接收接口，App 批量上传事件。
- 服务端再根据吞吐和消费需求选择 Redis Stream 或 Kafka：事件量较小、消费链路简单时可以用 Redis Stream；曝光和点击量很大、需要多系统订阅和长期回放时更适合 Kafka。

因此，本项目没有使用 Redis Stream 或 Kafka，并不是否定它们的价值，而是因为当前没有服务端、没有高吞吐和多消费者场景。用本地模拟消息队列更贴合 demo 目标，也让文档和代码重点仍然停留在广告流本身。

## 8. 效果评估

从功能完整性看，项目已经覆盖广告流的主要体验：多频道、多卡片类型、分页、筛选、详情、互动、AI 补全、搜索和埋点。即使不配置 LLM API Key，也可以依靠本地 fallback 跑通完整流程。

从列表体验看，分页加载和 DiffUtil 让刷新粒度比较可控；滚动状态按频道和标签保存，减少了切换后位置丢失的问题；视频播放器池限制了资源占用，不会因为多个视频卡片连续出现就无限创建播放器。

从 AI 稳定性看，项目没有把模型输出当作可靠接口，而是用 JSON 约束、清洗解析、字段 fallback、请求失败 fallback、in-flight 去重和 SQLite 持久化缓存共同兜底。这样 AI 内容可以逐步补齐，但不会阻塞广告列表展示。

从数据统计看，曝光口径有明确的停留时长和可见比例要求，且同时保留聚合计数和 telemetry 明细。聚合计数适合 UI 展示，明细表适合后续分析事件流。

## 9. 不足与后续优化

当前项目仍然是端侧示例，距离生产广告系统还有一些差距：

- AI 请求最好迁移到服务端代理，避免客户端直接持有 API Key，并方便统一做鉴权、限流、审核和日志脱敏。
- AI 缓存目前没有失效策略。后续可以根据广告内容版本、模型版本、提示词版本决定是否重新生成。
- 曝光去重目前只在 Fragment 生命周期内生效。生产环境通常还需要按用户、设备、广告、时间窗口做服务端去重。
- 曝光口径只计算垂直可见高度，没有进一步计算面积比例、遮挡关系和 App 前后台状态。
- telemetry 目前只落本地 SQLite，本质上是模拟消息队列，还没有上传链路、重试队列、批量上报策略和服务端消费者。
- AI 输出只做结构 fallback，还没有内容安全审核、敏感词处理和品牌合规校验。
- 分页策略是轻量本地分页，如果未来接入真实后端和大规模数据，可以考虑 Paging 3 或服务端游标分页。

整体来看，本项目的技术取舍是有意保持克制：不用过重框架解决小规模问题，也不为了演示 AI 而牺牲主流程稳定性。最终方案把广告流最容易出问题的几个点都收束在可维护的模块里，为后续扩展真实数据源、服务端 AI 和生产埋点留下了比较清楚的路径。
